## Finding 3 — mission-side `setSkill` layering vs ASR `CfgAISkill` — double-tuning risk map

**No mission-side `CfgAISkill` override exists.** A repo-wide grep for `CfgAISkill` across `.cpp`/`.hpp` in the mission and its addon folders (`wasp-tweaks/`, `Mods/`) returns zero hits. The skill-value **interpolation curve** (what a raw 0–1 `setSkill` number actually means for accuracy/reaction/spotting) is entirely owned by ASR AI's bundled config once `@adwasp` loads (Finding 1) — there is no mission-side config-layer contender.

**Mission code layers raw `setSkill` values on top, in at least four places**, all in `Missions/[55-2hc]warfarev2_073v48co.chernarus/`:

| Site | What it sets | When |
|---|---|---|
| `Common/Functions/Common_CreateUnit.sqf:57` | `_unit setSkill _skill` — scalar, defaults to `WFBE_C_UNITS_SKILL_DEFAULT` = **1** (`Common/Init/Init_CommonConstants.sqf:1712`), i.e. max on whatever curve is active | Every unit, at `createUnit` time — this is the universal baseline all AICOM/town/patrol units pass through |
| `Common/Functions/Common_CreateTownUnits.sqf:132-141` | Per-subskill randomized spread for **town garrison units only**: `aimingAccuracy` 0.65–0.95, `aimingSpeed`/`spotDistance` 0.70–0.95, `courage` 0.80–1.0, plus an overall scalar 0.80–1.0 | Once, at town-garrison spawn |
| `Common/Functions/Common_RunSidePatrol.sqf:72-80` | GUER-only (`_side == resistance`) near-max skill: scalar via `WFBE_C_SIDE_PATROL_GUER_SKILL` (default 0.85, tapering toward 1 as GUER town count drops), `aimingAccuracy` up to 0.95, `spotDistance`/`spotTime`/`courage` all pinned to **1** | Once, at side-patrol spawn |
| `Server/Functions/AI_Commander_Wildcard_GUER.sqf:423` | `general` skill bumped by `+0.05 * _tier` (tier-scaled garrison toughness) | Once, per garrison unit at composition-guard spawn |
| `Common/Functions/Common_RunCommanderTeam.sqf:84-90` | Optional "Veteran Company" scalar boost (0.85), gated on a trailing delegate arg only the AICOM-Teams HC dispatch path sends | Once, at team founding only — **not** re-applied on the recurring order/posture loop |

**Why this is a risk, not a confirmed bug:** every one of the writes above fires exactly once, at unit/team creation — none of them re-stamp skill on the recurring ~20s order cycle that `Common_RunCommanderTeam.sqf` uses for posture (`setCombatMode`/`setBehaviour`/`setSpeedMode`, `:97` and elsewhere). So there is no continuous per-tick fight between mission code and ASR over skill values, unlike the posture-stamp/mod-waypoint conflict class the AI-MODS-AND-PATHFINDING.md doc already analyzed for movement mods. The actual risk is narrower and twofold:

1. **Spawn-order race.** Both the mission's `Common_CreateUnit.sqf`/`Common_CreateTownUnits.sqf`/etc. scripts AND ASR's own local-unit-init skill/gear randomization (per ASR's design: *"configures only the AI local to the machine where it is installed"* — `AI-MODS-AND-PATHFINDING.md` §1(2)) fire off the same unit-created-locally event, on the same machine (the HC, for AICOM/town/patrol units). The engine does not guarantee ordering between a script-driven `createUnit` follow-up and an addon's own event-handler-driven local init. Whichever runs last for a given unit wins the *static* skill value for that unit — this recon found no instrumentation that would reveal which one currently wins, or whether it's even deterministic run-to-run.
2. **Category overlap without a value conflict.** Mission code pins categories like `courage`, `aimingAccuracy`, `spotDistance`/`spotTime` to specific numbers. ASR AI's danger.fsm independently and continuously modulates *behavior* tied to some of those same categories at runtime (documented in `AI-MODS-AND-PATHFINDING.md` §1(4): crouch/prone suppression reactions, "reluctant to move to waypoint" reports under fire). These are different mechanisms (a static skill number vs. a reactive FSM state), so they are not strictly "double-tuning" the same field — but the net behavior a player observes (e.g., a GUER patrol unit set to `courage=1` by mission code, then suppressed prone by ASR's danger.fsm reacting to the same `courage` category) is jointly produced by both systems, and neither this recon nor any prior doc has isolated which effect dominates.

**No live evidence either confirms or refutes an actual behavioral collision.** This finding characterizes the *mechanism* by which double-tuning could occur; it does not claim one has been observed. Resolving it would require in-engine instrumentation (e.g., logging a unit's `skill` array immediately after mission-side `setSkill` and again N seconds later) — an owner-gated live trial, not something derivable from static code.

## Finding 4 — HC delegation routing / load-balancing

**The picker: `Server/Functions/Server_PickLeastLoadedHC.sqf`.** Single source of truth for "which HC do we delegate to," used by every `delegate-aicom-team`/`delegate-townai`/`delegate-ai-static-defence`/`delegate-sidepatrol` dispatch site. Mechanism (full file read):

- Tallies `owner _x` over `allUnits` **once per call**, keyed against each live HC's `owner (leader _hc)`.
- Picks the **argmin** (currently-lightest HC by owned-unit count); ties broken **randomly** so the picker degrades to uniform-random when loads are equal and never lock-steps on the first-registered HC.
- Explicitly self-correcting by design (per its own header): *"Least-loaded reads CURRENT per-HC unit count every call... once an HC gets heavy it stops being chosen until the other catches up."* This directly supersedes an older blind `leader(_live select floor(random count _live))` coin-flip, which the header notes "NEVER self-corrects because delegated units are sticky (no migration)."
- Call sites: `AI_Commander_Teams.sqf:1138` (AICOM team founding — the "big atomic lump" case, per its own comment at `:1136`), `AI_Commander_Wildcard.sqf:723,862,1301,1524,1559` (W6/W19/W23/W24 wildcard dispatches), `server_side_patrols.sqf:274`, `Server_DelegateAIStaticDefenceHeadless.sqf:25`, and `Server_DelegateAITownHeadless.sqf:29` (hoisted — see perf note below).

**Known, already-fixed perf issue:** `Server_DelegateAITownHeadless.sqf:22-52` documents a **614ms town-activation spike** that was caused by calling the O(allUnits) picker **once per group** instead of once per town (O(groups × allUnits)). The fix: call the picker once per town-activation to find the currently-lightest HC, then locally round-robin that town's groups across all live HCs starting from that HC — same anti-pile-up spread, one scan instead of N.

**The gap: GUER wildcard garrison spawns bypass delegation entirely.** `Server/Functions/AI_Commander_Wildcard_GUER.sqf` (the composition-guard/garrison spawner, e.g. `:412-423`) creates its units via a direct `[_soldierClass, _grp, _gPos, _sideID] Call WFBE_CO_FNC_CreateUnit` call. A grep of that file for `PickLeastLoadedHC`, `delegate-aicom-team`, or `delegate-townai` returns **zero matches**. These units are created wherever `AI_Commander_Wildcard_GUER.sqf` itself executes — server-side, per its own commander-loop origin — and never routed to an HC. GUER garrison/composition-guard AI is, by design, **entirely server-local**; it never participates in the least-loaded-HC balancing that AICOM combat teams and delegated town AI use.

**Why this matters — it explains an open RPT question.** `docs/project-management/RPT-EVIDENCE-2026-07-05.md` (finding #2) flagged, as an open question requiring investigation: *"HC delegation degraded in the live 07-05 Chernarus session. `DELEGSTAT` remotePct fell from ~56% → 21% (t=775), partial recovery to 37% — vs 92–95% in both 07-04 sessions and 07-01 snapshot. Hypothesis: GUER + server-local groups dominate the AI budget while few towns are HC-delegated... Needs investigation before any HC/ASR tuning conclusions."* A follow-up soak reported in `docs/project-management/MORNING-REPORT-2026-07-06.md:8,43,50` found the same metric recovered to **89–95%** and offered a game-phase hypothesis (the 07-05 session was deeper/later-game with more active towns) but explicitly left it unconfirmed, recommending "a deep-game soak to confirm delegation holds under load." **The code-level fact found in this pass — GUER garrison spawns are unconditionally server-local and never counted as "delegated" — directly supports that hypothesis**: `DELEGSTAT remotePct` is `_delegRemote / _delegTotal` over `allUnits` (`server_groupsGC.sqf` DELEGSTAT block), so any phase of the match where GUER garrison/composition-guard activity is proportionally heavier (server-local, undelegated) relative to WEST/EAST AICOM+town activity (delegated) will mechanically depress the ratio — with no HC malfunction or build regression required to explain it.

## Finding 5 — how AICOM2/DECAP localizes execution onto the HC

**Strategic decisioning is 100% server-side.** `Server/AI/Commander/AI_Commander_Decapitate.sqf` — "AICOM v2 M5 DECAPITATE CLOSER. Server-side, per side" (its own header, `:2`) — runs after the M0 snapshot and M1 Allocator each strategy tick. It maintains an ARM→COMMIT→ABORT state machine gated by `WFBE_C_AICOM2_DECAP_ENABLE` (default **1**, `Common/Init/Init_CommonConstants.sqf:740`), with tuning constants also server-resident: `DECAP_DOM_RATIO` (1.5, `:741`), `DECAP_ABORT_RATIO` (0.9, `:742`), `DECAP_MAX_ENTOWNS` (5, `:743`), `DECAP_ARM_TICKS` (3, `:744`), `DECAP_MIN_COMMIT` (300s, `:745`), `DECAP_SENSE_RADIUS` (map-branched 2000/3000m, `:749`), `DECAP_SENSE_INTERVAL`/`DECAP_SENSE_CHANCE`/`DECAP_COMMIT_RADIUS` (`:750-752`). At `WFBE_C_AICOM2_DECAP_ENABLE = 0` it runs in shadow mode — sensing + telemetry only, **never writes the ORDER/stamp** — byte-identical to no-op.

**The only externally-visible effect is a broadcast group variable.** On COMMIT, the closer stamps `wfbe_aicom_decap` (a networked `setVariable [..., true]`, per its own header's "COMMIT ACTION" section) onto qualifying team groups whose leader is within `DECAP_COMMIT_RADIUS` of the sensed enemy HQ. It never touches `wfbe_aicom_targets` — the Allocator's town-first fist targeting stays authoritative for every team not currently stamped.

**Execution happens on whichever machine already owns that team's group — the HC, for AICOM teams.** `Common/Functions/Common_RunCommanderTeam.sqf:935` labels the consuming block the "DECAP PRESS HOOK (consumes the #724 organic-closer stamp; the first and only reader)." Each per-team driver pass (this file runs exclusively via the `delegate-aicom-team` dispatch, i.e. HC-locally per team, per `:179-182` "HC locality note") reads `_team getVariable "wfbe_aicom_decap"` (`:940-941`) and, while stamped, overrides that pass's destination/mode with the stamped HQ position via the existing "goto"/press movement idiom (`:942-957`) — fixed-wing teams are excluded from the press (`:942`, ground-press only in this increment). No separate HC-side DECAP logic exists; the HC-local per-team driver is simply reading a state flag the server computed and broadcast, exactly the same networked-groupVar pattern the rest of the AICOM order pipeline (`wfbe_aicom_order`) already uses.

**Verdict:** this is a clean, low-coupling "compute centrally, execute locally" split — the strategic decision never needs to know or care which HC (or the server, as fallback) owns a given team; it only needs the team's group object to carry the stamp, and the HC-local driver already reads that team's own vars every pass regardless of origin.

## Finding 6 — `groupsGC` / server-loop perf

**Two GC loops, split by locality, not one.**

- **`Server/FSM/server_groupsGC.sqf`** — `if (!isServer) exitWith {}` (`:8`) — server-local groups only. `while {!WFBE_GameOver} do { sleep 60; ... }` (`:18-19`) reaps zero-living-unit, non-persistent groups every 60s against A2's hard ~144-groups/side cap. Debounced (5 min, `_warnInterval = 300`, `:14`) group-cap pre-warnings fire at `>= 130` (approach) and `>= 144` (at-cap) per side (`:394-436`).
- **`Client/Functions/Client_GroupsGC.sqf`** — runs on player clients (`hasInterface`) **and**, since 2026-06-15, on headless clients (`isMultiplayer && !isServer && !hasInterface`, `WFBE_GC_IsHC` at `:19`) specifically because HC-local empty-group husks are invisible to the server sweep: `deleteGroup` only reaps a group that is both empty AND local to the executing machine, so `server_groupsGC.sqf` silently no-ops on HC-owned empties. The file's own header explains the fix history: an HC-owned commander team's self-reap (`Common_RunCommanderTeam.sqf:789 deleteGroup _team`) can no-op if dead-but-not-yet-engine-collected corpses still sit in `units _team`; the resulting empty husk is HC-local and, before this broadening, nothing reaped it — a real leak toward the 144/side cap that this fix closed.

**A real correctness bug was fixed in the server-side sweep.** `server_groupsGC.sqf:25-45` (`cmdcon41-w3c`) replaced a single-pass `forEach` that called `deleteGroup` inline while iterating `allGroups` — A2 OA 1.64 leaves `forEach` behavior undefined when the collection is mutated mid-iteration (the same trap already fixed once before in `Common_CreateGroup.sqf:36-54`, per the inline comment) — with a two-pass collect-candidates-then-delete pattern, closing a contributor to "observed transient empty-group pile-up."

**Diagnostic cost is throttled, not free-running.** The expensive per-faction group-classification "audit dump" (per `Common/Init/Init_CommonConstants.sqf:1885`, measured **~2100ms on 276 groups**) is gated behind `WFBE_C_GROUPAUDIT_EVERY` (default **5**) so it fires roughly every 25 minutes instead of every 5-minute audit window; the always-on husk-reap + zombie-reap + cap-warning passes remain cheap per-tick regardless of the audit gate.
