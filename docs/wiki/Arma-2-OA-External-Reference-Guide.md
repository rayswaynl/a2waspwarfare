## Guardrails

- `publicVariable` broadcasts the current missionNamespace variable value; later local changes do not sync automatically unless code broadcasts again. The publicVariable page also notes that broadcast variables are available to JIP clients with their last broadcast value. That is last-value state, not event-history replay or full collection sync; marker sets, queues and other feature state still need explicit server-held state plus resend or request/response.
- `addPublicVariableEventHandler` fires when a broadcast is received. The current BI page includes Arma 3 deprecation and alternative-syntax notes, so for OA 1.64 use the source-backed Wasp pattern: handlers consume the broadcast variable name/value and do not expose a trusted sender identity. Wasp server handlers must re-derive side, role, funds, object ownership and range from server-owned state.
- `publicVariableServer` and `publicVariableClient` target direction; they do not validate the gameplay authority of the payload. This is why DR-1 PVF dispatcher hardening and DR-41 direct-channel hardening are separate layers.
- `isServer`, `isDedicated` and `hasInterface` describe different machine roles. In this mission, hosted/player-server paths are intentional, and headless clients are excluded through `isHeadLessClient` / `hasInterface` checks. Do not simplify role gates to dedicated-only logic unless the source path is truly dedicated-only.
- `local` and `owner` are routing/locality primitives, not authority proofs. Use `local` before mutating AI/vehicle state where the engine requires object locality, and use `owner _player publicVariableClient` for targeted delivery; still validate gameplay authority in server-owned state.
- `paramsArray` is index-aligned to `missionConfigFile >> "Params"` order in this mission. Keep `Rsc/Parameters.hpp` class order stable, and do not confuse mission config `class Params` with the A3-only SQF `params` command.
- Config lookups use OA-safe `configFile` / `missionConfigFile`, `>>`, typed getters and manual `count`/`select` scans. Do not copy Arma 3 `getMissionConfigValue`, `configClasses`, `configProperties`, `apply`, or `#include "\a3\..."` mission-parameter template examples into OA mission SQF.
- `nearestObjects` accepts a class-name array and `[]` searches all classes. The command uses `isKindOf` matching, so a parent class such as `Base_WarfareBUAVterminal` can cover inherited command-center terminals.
- Object lifecycle commands are split by network intent: use `createVehicle` / `createUnit` for real mission objects, `createVehicleLocal` for client-only previews/FX, and OA `setVehicleInit` / `processInitCommands` for old JIP-visible init. Do not import A3 `createVehicleCrew`, `deleteVehicleCrew`, `hideObjectGlobal`, `enableSimulationGlobal` or lifecycle-page `remoteExec` examples into OA mission SQF.
- `preprocessFileLineNumbers` preserves file/line context for runtime errors. Prefer it for compiled mission files, and do not reintroduce per-message `Call Compile` on sender-controlled strings.
- `spawn`, `execVM` and `execFSM` create scheduled script/FSM work with handles; `call` executes in the current context. Keep the PVF dispatcher scheduling shape unless each handler has been audited for sleeps/waits, and use `scriptDone`/`terminate` only with stored handles that also have cleanup paths.
- `waitUntil` in OA is the code-condition form. Add `sleep`/`uiSleep` to polling waits where possible, and do not copy Arma 3 `waitUntil [condition, timeout, interval]` or `waitUntil scriptHandle` examples into mission SQF.
- `hasInterface` is OA 1.63+ per BI's command page and A2OA patch notes; this fork uses it in the HC detector and performance-scope labels. Treat it as source-backed for OA 1.64, but do not copy unrelated Arma 3 locality examples into mission SQF.
- Avoid Arma 3-era assumptions unless explicitly labelled as non-authoritative. In particular, do not introduce `remoteExec`, `remoteExecCall`, SQF `params`, `parseSimpleArray`, `isEqualTo`, `setGroupOwner`, `groupOwner`, modern `select [start,count]` / `select {condition}` forms or `private _var = value` into OA mission SQF without an OA availability proof. `BIS_fnc_MP` is documented for Take On Helicopters/Arma 3 rather than the OA function category, and this repo already has its own PVF/publicVariable wrappers.
- Do not import A3 remote-execution identity helpers (`remoteExecutedOwner`, `isRemoteExecuted`) or A3 `allPlayers` examples into OA code. Wasp uses PVEH payloads, `owner`-targeted public variables, `isPlayer`/`playableUnits`, and explicit server-side validation instead.
- Also avoid `compileFinal`, `canSuspend` and modern script-handle/wait syntax from Arma 3 examples. They are contrast material only for this OA 1.64 mission.
- Some BI pages mix OA primitives with newer Arma 3 examples. Use the page's game/version icons, the [Arma 2 OA compatibility audit](Arma-2-OA-Compatibility-Audit) and the [Arma 2 OA command version reference](Arma-2-OA-Command-Version-Reference) before copying syntax.
- Object/display/control event handlers are OA-safe, but they stack unless removed or guarded. Reusing a vehicle or mission object and adding another handler can multiply side effects; current source has this class in `supplyMissionStarted.sqf:11` and duplicate `Fired`/`HandleReload` wiring in `Client_BuildUnit.sqf:332-334`.
- UI display/control handlers are client-local. Wait for the display/control, use `disableSerialization` when storing UI handles, throttle display polling and check IDD collisions before relying on `findDisplay`.
- Local marker commands are correct for per-client map/HUD rendering. They are not global state or automatic full JIP state. Pair `createMarkerLocal` with local setters and `deleteMarkerLocal`; if reviving dead marker networking, keep server-held marker lists and client-local rendering as separate layers. When global marker names are required for late joiners, follow the distinct `Common_CreateMarker.sqf` model: global marker creation plus local visual properties, not accidental global mutation of a local marker.
- Treat `addMissionEventHandler` as an A3 contrast warning unless an OA source proof appears. Current Wasp source has no live `addMissionEventHandler` use; use existing object, MP, display/control and public-variable patterns instead.

## Common Mistakes

| Mistake | Better move |
| --- | --- |
| Treating a client menu check as server authority. | Keep the menu as affordance and validate the request on the server. |
| Assuming a public-variable handler proves who sent a request. | Reconstruct requester/side from trusted server state or channel context. |
| Replacing a broad `nearestObjects` scan without smoking gameplay. | Use the class-filter candidate, then smoke real map objects and mission behavior. |
| Copying an Arma 3 wiki example directly into OA code. | Check OA command availability and prefer source-local idioms already used by this repo. |
